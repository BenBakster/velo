# velo — порядок операций (канон последовательности)

Единая карта «что за чем»: **собрать → проверить → записать → загрузить → установить**.
Это **оглавление потока**, а не замена детальным рунбукам — каждый шаг ссылается на свой
подробный документ. Если детали в рунбуке и здесь разойдутся — **правда в рунбуке**, а этот
файл синхронизируется по нему.

## Прежде всего — где мы и правила

**Прочитать перед любым шагом:**
- `docs/HANDOFF.md` — текущее состояние installer-арки (M0–M4) и что закрыто.
- `docs/SESSION-PLAN-2026-06-12.md` — актуальное направление v0.2 (профиль **homely**) + acceptance §4.3.
- `PLAN.md` — решения Ворот 0–2, милстоуны, ограничения.
- `docs/constraints.md` — ограничения ramdisk/TUI (почему ASCII, нет `printf`/`awk`/`tput`).

**Стоячие правила (не нарушать автономно):**
- **`sda` / `nvme0n1` — НИКОГДА не цель записи.** `sda` = Void-portable клон, `nvme0n1` =
  система Void. Цель опознавать **по содержимому** (removable + `TRAN=usb` + размер в окне),
  а не по букве `sdX`.
- **Реальные диски — только supervised** (Ворота-СТОП, при живом Антоне).
- **Push — только по явному «пуш»** от Антона. Локальные коммиты — свободно.
- **Правки в репо — только `sudo -u thx1138`**, не из-под root (иначе ломается владелец файлов).

## Поток

### Шаг 0 — (Пере)сборка образа `dist/velo79.img`
**Когда:** образа нет, либо он отстал от текущего `HEAD` (код сборки и `site/` ушли вперёд).
**Где:** внутри **OpenBSD 7.9 build-VM** (`vm/buildhost.qcow2`) — `build-velo`/`assemble-media`
требуют OpenBSD (`rdsetroot`/`vnconfig`). Транспорт в VM — `vm/ssh_pw.py` (`root@127.0.0.1:2222`).
**Цепочка:** `make-site-tgz.sh` → `build-velo.sh` (патч `bsd.rd`) → `assemble-media.sh` →
`dist/velo79.img` **+ сайдкар `dist/velo79.img.sha256`** (его эмитит сборка).
**Текущий образ:** профиль **homely** (openbox+tint2 + xterm/terminator + Firefox + кириллица
в xterm), собран 2026-06-13 (closure-fix). **Контрольные суммы не хардкодим — сверка по сайдкару.**
**Перед сборкой — зелёные тесты:** `ksh src/velo-install selftest`, `tests/*` (счётчики — в `HANDOFF.md`).
**Детали:** `docs/m3-runbook.md` §1–§3 + `docs/WORKLOG.md` («VM-сборка»).

### Шаг 1 — Приёмка в VM (ничего реального не пишется)
**qemu против чистого пустого диска:**
- TUI рендерится (синий ASCII-бокс, без стен `?`) и появляется через хук `/.profile` —
  `docs/m3-runbook.md` §4.
- Полная зашифрованная установка + матрица приёмки (FDE-prompt, pf L1/L2/L3, doas,
  offline-closure, L3 SOCKS-only fail-closed) — `docs/m3-runbook.md` §5.
- **homely acceptance:** serial/qemu ✅ (`bash vm/homely-accept.sh`); VGA/визуал (кириллица
  в xterm, панель/меню) и metal — ⬜ pending (`docs/SESSION-PLAN-2026-06-12.md` §4.3,
  `vm/homely-vga-accept.py`, supervised).

### Шаг 2 — Запись на USB ⛔ HARD STOP (supervised, только Антон)
**Необратимо. Никакой автоматики, никакого «угадаем диск».**
- Сверь образ по сайдкару `dist/velo79.img.sha256` (скрипт сверяет сам до `dd`; mismatch = отказ).
- Опознай цель: removable + `TRAN=usb` + размер в окне; корень НЕ на цели; **не `sda`, не `nvme`**.
- Void/Linux: `build/flash-sda-guarded.sh` (dry-run по умолчанию, typed-confirm, post-write `cmp`).
- OpenBSD: `build/write-usb.sh`.
**Детали:** `docs/usb-write-runbook.md` (полный guard-стек + карта дисков + Ворота-СТОП).

### Шаг 3 — Загрузка целевого ноута с USB
Воткнуть USB в целевую машину; в BIOS/boot-menu выбрать загрузку с USB (legacy/SeaBIOS **или**
UEFI — velo поддерживает оба, мастер сам определит прошивку). Появляется ASCII-TUI velo.

### Шаг 4 — Установка на железо ⛔ HARD STOP (supervised)
Мастер (≈9 экранов при FDE): диск → шифрование → пароль(и) → hostname → **профиль = homely** →
пакеты → Protection level (L1 baseline / L2 / L3 Tor-only) → boot-mode (bios/uefi/hybrid) →
summary → **DESTROY-гейт** (впечатать имя диска).
- **Целевой диск установки — ОТДЕЛЬНЫЙ** (рекоменд. внешний SSD), **не сам USB и не `nvme`**.
  Мастер до деструктива покажет идентичность диска (размер/DUID); **homely требует ≥ 28 GiB**.
- Установка оффлайн; на первой загрузке `pkg_add` ставит homely-стол из встроенного closure,
  дальше — xenodm → openbox+tint2.
**Детали:** `docs/m3-runbook.md` §6 (M4) + `docs/usb-write-runbook.md` §4.

## Если что-то не так
- **Образа нет / сверка `.sha256` не прошла** → Шаг 0 (пересборка из `HEAD`; цифры — в сайдкаре).
- **Не грузится с USB** → проверь boot-order в BIOS; для очень старого железа в мастере есть
  boot-mode `bios` (это про **целевой** диск, не про USB).
- **Сомнение в цели записи → СТОП**, переспросить. Лучше переспросить, чем затереть `sda`/`nvme`.
